## Problem Analysis

### Issue Reported
Several frame types failed to render OG images when shared on Farcaster feed:
- ❌ Badge Collection frames (`/frame/badge/18139`)
- ❌ Leaderboard frames (`/frame/leaderboard`)
- ❌ Points Balance frames (`/frame/points?fid=18139`)
- ❌ Onchain Stats frames (`/frame/stats/18139`)

### Working Frames (for comparison)
- ✅ GM Streak (`/frame/gm?fid=18139`)
- ✅ Guild (`/frame/guild?fid=18139`)
- ✅ Referral (`/frame/referral?fid=18139`)

## Root Cause Analysis

### Why It Failed

The failing frame routes were using **internal fetch() calls** to proxy requests to `/api/frame`:

```tsx
// PROBLEMATIC PATTERN (before fix)
const frameUrl = new URL('/api/frame', origin)
frameUrl.searchParams.set('type', 'badge')
frameUrl.searchParams.set('fid', String(fid))

const frameResponse = await fetch(frameUrl.toString(), {
  headers: { 'User-Agent': req.headers.get('User-Agent') || 'Farcaster-Crawler/1.0' }
})

const frameHtml = await frameResponse.text()
return new NextResponse(frameHtml, { ... })
```

**Problem:** When Farcaster crawlers fetch the initial route (e.g., `/frame/badge/18139`), they expect the HTTP response to contain OG meta tags. But the internal fetch approach meant:

1. Crawler fetches `/frame/badge/18139`
2. Route handler internally fetches `/api/frame` to get HTML
3. HTML is returned to crawler in response body
4. But crawlers don't execute JavaScript or follow redirects - they just read the initial response headers

The OG tags were embedded in the fetched HTML body, but crawlers extract them from the initial HTTP response.

## Solution Implemented

### Direct Handler Calls

Changed all frame routes to **directly call their respective handlers** instead of proxying:

#### 1. Badge Frame Route
**File:** `/app/frame/badge/[fid]/route.tsx`

```tsx
// FIXED PATTERN (after fix)
import { handleBadgeFrame } from '@/lib/frames/handlers/badge'

export async function GET(req: Request, { params }) {
  // ... validation ...
  
  // Call handler directly
  const response = await handleBadgeFrame({
    req, url, params: { type: 'badge', fid: String(fid), ... },
    traces: [], origin, defaultFrameImage, asJson: false,
  })
  
  // Return response directly with proper headers
  return new NextResponse(response.body, {
    status: response.status,
    headers: { 
      ...Object.fromEntries(response.headers.entries()),
      'Content-Type': 'text/html; charset=utf-8',
      'Cache-Control': 'public, max-age=300, s-maxage=300, stale-while-revalidate=60',
    },
  })
}
```

#### 2. Onchain Stats Frame Route
**File:** `/app/frame/stats/[fid]/route.tsx`

Same pattern - calls `handleOnchainStatsFrame()` directly

#### 3. Leaderboard Frame Route
**File:** `/app/frame/leaderboard/route.tsx`

Same pattern - calls `handleLeaderboardFrame()` directly

#### 4. Points Frame Route
**File:** `/app/frame/points/route.tsx`

Same pattern - calls `handlePointsFrame()` directly

## Why This Works

When a frame handler is called directly:

1. Handler generates HTML with OG meta tags (e.g., `<meta property="og:image" content="..." />`)
2. HTML is returned with proper headers set
3. Route returns this response directly to the crawler
4. Crawler sees OG tags in the initial HTTP response ✅

The frame handlers already had all the proper OG tag generation code, it just wasn't being delivered correctly.

## Technical Details

### Frame Handler Response Structure

Each handler returns a Response object that includes:
- HTML body with OG meta tags
- Proper Content-Type header
- Cache-Control headers for performance

Example OG tags generated by handlers:
```html
<meta property="fc:frame" content="vNext" />
<meta property="fc:frame:image" content="https://gmeowhq.art/api/frame/image/badge?..." />
<meta property="fc:frame:image:aspect_ratio" content="1:1" />
<meta property="og:image" content="https://gmeowhq.art/api/frame/image/badge?..." />
<meta property="og:title" content="Gmeowbased - Badges" />
<meta property="og:description" content="X badges earned" />
```

### Cache Headers

All routes now use improved cache headers:
- `public, max-age=300` - Cache for 5 minutes
- `s-maxage=300` - Shared cache (CDN) for 5 minutes
- `stale-while-revalidate=60` - Serve stale content while revalidating

## Lessons Learned

1. **Farcaster Crawlers** don't follow internal redirects or fetches - they need OG tags in the initial response
2. **Direct Handler Calls** are more reliable than proxying for frame generation
3. **Response Headers** matter - proper Cache-Control ensures Farcaster caches the rendered images
4. **Testing Crawlers** requires actual Farcaster testing, not just local preview
